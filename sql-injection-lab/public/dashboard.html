Testing Guide (Burp Workflow)

              
High-level, defensive testing steps. Do not test systems without permission.

            

            

              


              

                

                  

                    
/login

                    AUTH
                  

                  

                    1) Intercept request  2) Send to Repeater  3) Change fields slightly  4) Compare status codes/response sizes. Observe
                    cookie flags and error messaging.
                  

                

                

                  

                    
/api/search

                    SQL
                  

                  

                    1) Intercept search request  2) Replay with unusual text  3) Compare responses between Demo and Secure mode. Observe
                    whether the server returns safe, consistent results.
                  

                

                

                  

                    
/profile?id=

                    AUTHZ
                  

                  

                    1) Request your profile  2) Change the id value  3) Compare the response. Observe whether authorization is enforced
                    server-side (it should be).
                  

                

                

                  

                    
/api/comments

                    XSS
                  

                  

                    1) Submit a comment  2) Replay with special characters  3) Observe safe encoding and CSP behavior. Note: VulnLab does not
                    execute scripts.
                  

                

                

                  

                    
/admin

                    ADMIN
                  

                  

                    1) Access as non-admin  2) Observe access denied  3) Access as admin  4) Observe stats and event telemetry.
                  

                

              


              

                What to observe: status codes, response length changes, safe error messages, authorization behavior, and differences between Demo
                and Secure mode.

Learning Center

              
What it is, why it works, impact, and how to fix it (secure coding).

            

            

              


              

                

                  
SQL Injection

                  
Unsafe vs safe query building.

                

                

                  
Why it works: concatenation lets input change the structure of the query.

                  
UNSAFE (example only):
const q = "SELECT * FROM users WHERE u='" + user + "' AND p='" + pass + "'";

SAFE:
db.get("SELECT * FROM users WHERE u=? AND p=?", [user, pass]);

                  
Prevention: prepared statements, validation, least privilege, and monitoring.

                

              


              

                

                  
XSS

                  
Stored, reflected, and DOM-based concepts.

                

                

                  
Impact: session theft, UI redress, malicious actions as the user.

                  
SAFE RENDERING:
// Use textContent (or server-side encoding) for untrusted input.
el.textContent = userInput;

DEFENSE-IN-DEPTH:
Content-Security-Policy (CSP) + output encoding + avoid dangerous sinks.

                

              


              

                

                  
Authentication + Sessions

                  
Weak vs strong authentication and session handling.

                

                

                  
GOOD PRACTICE:
- Hash passwords (bcrypt/argon2)
- Use HttpOnly + SameSite cookies
- Expire sessions, rotate identifiers, invalidate on logout

                

              


              

                

                  
IDOR (Access Control)

                  
Authorization checks on every object access.

                

                

                  
SAFE PATTERN:
// Never trust the id from the URL alone.
// Enforce ownership/role checks server-side for each request.

                

              

            

          


          

            

              
Defense & Protection

              
Practical defensive guidance you can apply to real projects.

            

            

              


              

                

                  

                    
Input Validation

                    APP
                  

                  

                    Validate types, lengths, and allow-lists. Treat validation as data-shaping, not security alone. Pair with parameterization.
                  

                

                

                  

                    
Secure Authentication

                    AUTH
                  

                  

                    Hash passwords (bcrypt/argon2), rate-limit, add MFA, and monitor for credential stuffing. Use safe error messages.
                  

                

                

                  

                    
Authorization

                    AUTHZ
                  

                  

                    Enforce ownership/role checks server-side for every object access. Never trust IDs from the URL alone.
                  

                

                

                  

                    
Output Encoding + CSP

                    XSS
                  

                  

                    Render untrusted input as text, avoid dangerous sinks, and use a strict CSP to reduce XSS impact.
                  

                

                

                  

                    
Network Hardening

                    NET
                  

                  

                    Reduce exposed services, segment networks, apply firewall rules, and keep an inventory of what's reachable.
                  

                

                

                  

                    
Logging & Monitoring

                    OBS
                  

                  

                    Log security-relevant events, detect anomalies, and alert. Correlate auth, input flags, and access denials.
